三阶段已经完成，并且当前仓库状态满足进入四阶段的前置条件。

本次检查结果如下：

- 三阶段验收记录中声称的关键实现，已在当前代码中核对到：
  - `desktop_chat_gui.py` 已包含连接状态锁定、联系人指纹展示、彩色 Crypto Console、断连清理和自动解密展示逻辑。

因此，四阶段不再继续扩展桌面端最低合规版的主链路，而是转入“**交付收口与验收包装阶段**”。

四阶段的核心判断是：

1. 三阶段解决了“程序能不能稳定完成端到端加密聊天”的问题。
2. 四阶段要解决“项目能不能被别人稳定复现、打包演示、写进报告并按课程要求提交”的问题。

## 2. 四阶段目标

四阶段的唯一主目标，是把当前已经完成的桌面端最低合规版，收敛为一个**可复现、可打包、可写报告、可做汇报、可最终提交**的课程设计交付物。

四阶段完成后，项目应达到以下状态：

1. 新环境按文档安装依赖后，可以稳定运行服务端、桌面客户端，并通过全量自动化测试。
2. 当前聊天主程序有明确的打包命令、打包说明和最小运行验证步骤，而不是只保留旧的 RSA 单机工具产物。
3. 报告骨架、实验结果说明、原创性说明、汇报脚本和最终提交检查单均已落地为仓库文件。
4. 抓包验证、截图清单和演示流程形成可复现材料，便于课程答辩和最终报告取证。
5. 项目最终提交所需的源码、可运行程序、实验结果说明和命名规则都已经收口，不再依赖临场整理。

## 3. 四阶段范围

### 3.1 本阶段要完成的内容

- 收口依赖安装、环境初始化和测试验证流程。
- 收口桌面聊天主程序的打包方案与运行说明。
- 补齐课程硬性要求对应的文档与提交物骨架。
- 补齐抓包验证、截图清单、汇报脚本等演示材料。

- 不做文件传输开发。
- 不做群聊、消息持久化、复杂账号系统等新功能。
- 不重构已稳定的加密、协议、服务端和桌面主链路，除非发现阻塞交付的缺陷。

说明：

- 当前课设已经达到最低合规版标准，四阶段应优先保证交付成功，而不是继续把范围扩展回增强功能。

## 4. 四阶段开发顺序

### 任务 A：收口环境初始化与依赖复现

目标：确保新的评审环境或答辩机器，不会再出现“代码存在但依赖未装齐，导致集成测试无法运行”的问题。

涉及模块：

- `requirements.txt`
- `pyproject.toml`
- `README.md`
- `document/uv_usage.md`
- `tests/manual_acceptance.md`

必须完成：

- 固定一条最小可复现安装路径，例如 `pip install -r requirements.txt` 或 `uv sync`。
- 在 README 中补齐项目简介、依赖安装、服务端启动、客户端启动、测试命令和常见问题说明。
- 在 `tests/manual_acceptance.md` 中统一依赖安装口径，避免文档之间安装方式不一致。
- 明确记录 `websockets` 属于必需依赖，不能只依赖本地环境残留状态。
- 固定“安装依赖 -> 运行测试 -> 启动服务端 -> 启动双客户端”的标准验证顺序。

完成标准：

- 一个新环境只看仓库文档即可完成依赖安装。
- `python -m unittest discover -s tests -v` 可稳定通过。
- 不再出现因为遗漏依赖而误判项目不可运行的情况。

### 任务 B：收口桌面聊天主程序打包链路

目标：把当前的聊天主程序变成真正可交付的“可运行程序”，而不是只保留源码或旧的 RSA 演示工具产物。

涉及模块：

- `desktop_chat_gui.py`
- `chat_server.py`
- `README.md`
- 打包配置文件（可为 `.spec` 文件或脚本化命令说明）

必须完成：

- 明确当前课程主程序入口是 `desktop_chat_gui.py`，而不是旧的 `InfoSecurWork_GUI.py` 单机工具。
- 为桌面聊天客户端提供 PyInstaller 打包命令，并写清输出目录和运行方式。
- 至少验证一次桌面聊天客户端打包结果可启动。
- 如成本可控，同步为 `chat_server.py` 提供控制台版打包命令，降低演示机缺少 Python 环境时的阻力。
- 在 README 或结果说明中明确区分：旧的 `dist/RSA_Encrypt_Decrypt_Tool.exe` 是历史工具产物，不是当前课设主交付程序。

完成标准：

- 当前聊天客户端至少存在一条已验证的可打包路径。
- 仓库中有清晰的打包说明，老师或组员不需要反向推测主程序入口。

### 任务 C：补齐报告骨架与原创性说明材料

目标：把课程要求中的“最终报告结构”和“原创性说明”提前落地，避免最后阶段只剩复制粘贴式整理。

涉及模块：

- `document/report_outline.md`
- `document/result_note.md`
- `document/design_decisions.md`
- `document/team_division.md`

必须完成：

- 按任务书要求建立固定报告骨架，章节必须包含：任务分析、理论基础、解决思路与设计、实现与测试、结果分析、运行截图、程序使用说明、团队分工。
- 补写实验结果说明，明确运行环境、启动方式、测试结果、异常处理和演示结论。
- 补写原创性说明，至少覆盖以下决策：为什么使用 RSA + AES-GCM、为什么服务端采用盲转发、为什么桌面端使用 Crypto Console、为什么保留 `InfoSecurWork_GUI.py` 作为旧工具而另建聊天主程序。
- 如果是个人项目，在 `team_division.md` 中明确标注“本项目为个人完成”；如果是小组项目，则分模块记录分工和完成情况。

完成标准：

- 报告骨架已经可直接填充，不需要在最终周重新设计结构。
- 原创性材料足以支撑“我理解并主导了实现决策”的答辩说明。

### 任务 D：补齐汇报脚本、截图清单与抓包证明材料

目标：把“能讲清楚”和“能证明传输的是密文”这两件事提前做成固定材料。

涉及模块：

- `document/week8_demo_script.md`
- 抓包说明文档（建议新建到 `document/` 下）
- `tests/manual_acceptance.md`

必须完成：

- 编写 15 分钟汇报脚本，固定理论部分、代码展示顺序、运行演示顺序和关键讲解句式。
- 明确截图清单，区分 GUI 截图、服务端日志截图、抓包截图和异常场景截图。
- 补写 Wireshark 抓包步骤，至少明确过滤条件、演示端口、截图节点和要点说明。
- 把“服务端看不到明文”的证据口径固定下来，确保文档、汇报和报告中的表述一致。

完成标准：

- 按脚本演示一次即可覆盖理论、代码、结果说明三部分。
- 抓包验证路径清晰，后续只需要执行和截图，不需要再临场摸索。

### 任务 E：完成最终提交检查单与交付彩排

目标：把课程最终提交动作变成一个可勾选的流程，而不是依赖最后一天人工回忆。

涉及模块：

- `document/submission_checklist.md`
- `README.md`
- `tests/manual_acceptance.md`

必须完成：

- 固定最终提交物清单：最终报告、程序源码、可运行程序或运行说明、实验结果说明。
- 固定压缩包命名规范“第X组+题目”。
- 固定提交邮箱 `xiongf@bjtu.edu.cn`。
- 做一次交付彩排：从依赖安装、测试、运行、截图、打包、文档检查一路走完。
- 记录彩排中发现的缺口，例如命令不一致、路径错误、截图缺项、文档空白等，并及时修正。

完成标准：

- 最终提交动作可按清单逐项勾选完成。
- 到正式提交时，不再存在“材料散落在各处、命名不统一、步骤靠口头记忆”的问题。

### 任务 F：完成四阶段总体验收

目标：确认项目已从“开发完成”进入“可答辩、可提交、可复现”的状态。

涉及模块：

- `README.md`
- `requirements.txt`
- `tests/manual_acceptance.md`
- 打包输出物
- `document/` 下新增或补齐的所有交付材料

必须完成：

- 再次运行全量自动化测试并记录结果。
- 按人工验收手册完成一次双客户端演示。
- 按打包说明完成一次程序打包验证。
- 核查 README、实验结果说明、报告骨架、汇报脚本和提交检查单是否齐备。

完成标准：

- 项目从代码、运行、打包、报告到提交检查单形成闭环。
- 四阶段结束后，剩余工作只应是填充截图、撰写正文和整理最终压缩包，而不是再补关键流程。

## 5. 四阶段建议优先级

### 第一优先级：先把交付可复现性做实

- 依赖安装说明。
- README 快速启动。
- 全量测试验证。
- 打包命令和主程序入口确认。

### 第二优先级：再把课程硬性文档补齐

- 报告骨架。
- 实验结果说明。
- 原创性说明。
- 团队分工。
- 提交检查单。

### 第三优先级：最后做演示证明材料

- 汇报脚本。
- Wireshark 抓包步骤。
- 截图清单与答辩话术。

说明：

- 如果第一优先级未完成，不应提前扩展 Web 端或文件传输。
- 如果前两级均稳定完成，再评估是否追加少量加分包装内容。

## 8. 阶段边界说明

四阶段的重点，是把“已完成的程序”收口成“可以最终交付的课设作品”。

如果四阶段重新转向以下内容，应视为越界：

- Web 端互通开发。
- 文件传输开发。
- 群聊或历史消息功能开发。
- 对现有协议、加密主链路做大幅重构。

这些内容不会提升当前阶段的交付成功率，反而会重新引入实现风险。

## 9. 最后结论

从当前检查结果看，三阶段已经完成，项目的“最低合规版桌面端到端加密聊天程序”已经成立。

四阶段不应该再把精力投入到高风险的新功能，而应该集中完成以下三件事：

1. 把环境、依赖、测试和打包做成可复现闭环。
2. 把报告、汇报、实验说明和提交检查单补齐为仓库文件。
3. 把抓包、截图和演示流程固化，确保答辩和最终提交不失控。

只要四阶段按本计划执行，项目就会从“代码完成”进入“真正可交付”的状态。
