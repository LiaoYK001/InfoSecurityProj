## 环节二：理论部分（4 min）

### 2.1 加密算法（2 min）

**讲解要点**：

- AES-GCM：对称加密，256-bit 密钥，12-byte Nonce，AEAD 模式同时提供加密和完整性保护。
- RSA-OAEP(SHA-256)：非对称加密，用于安全传输一次性 AES 会话密钥。
- "为什么不直接用 RSA 加密消息？—— RSA 有长度限制，1024-bit 密钥最多加密 86 字节。"

### 2.2 混合加密方案（1 min）

**讲解要点**：

- 加密链路：明文 → 生成 AES-256 密钥 → AES-GCM 加密 → RSA-OAEP 包裹会话密钥 → 发送
- 解密链路：收到密文 → RSA 解包会话密钥 → AES-GCM 解密 → 明文
- "每条消息使用独立的一次性会话密钥，保证前向安全性。"

### 2.3 盲转发设计（1 min）

**讲解要点**：

- 服务端只做消息路由，不持有私钥，无法解密。
- "即使攻击者控制了服务端，也无法获取聊天内容——这就是端到端加密的核心价值。"

### 3.2 关键代码走读（3 min）

**展示文件和要点**：

1. **`message_crypto.py` → `encrypt_chat_message()`**：
   - "这是加密的入口函数。可以看到先生成 AES 密钥，再 AES-GCM 加密正文，最后 RSA 加密会话密钥。"

2. **`aes_core.py` → `encrypt_text()`**：
   - "AES-GCM 加密，12 字节随机 Nonce，返回 Base64 编码的密文和 Nonce。"

3. **`chat_server.py` → `_handle_chat_message()`**：
   - "服务端收到消息后，只检查 receiver_id 并转发，完全不碰 payload。日志只记录 payload 长度。"

**可能的提问与答案**：

| 提问                              | 回答要点                                             |
| --------------------------------- | ---------------------------------------------------- |
| 为什么用 AES-GCM 而不是 AES-CBC？ | GCM 是 AEAD 模式，自带完整性校验，无需额外 HMAC      |
| 服务端真的看不到明文吗？          | 是的，可以看服务端日志和抓包，payload 全是密文       |
| 密钥怎么交换的？                  | RSA 公钥通过服务端转发，会话密钥用 RSA-OAEP 加密传输 |
| 前向安全性是什么意思？            | 每条消息独立会话密钥，一条泄露不影响其他             |

## Wireshark 抓包步骤

### 准备

1. 安装 Wireshark（https://www.wireshark.org/）
2. 启动 Wireshark，选择 Loopback 网络接口（本机通信使用回环接口）

### 捕获

1. 设置过滤条件：`tcp.port == 8765`
2. 点击开始捕获
3. 按正常流程启动服务端和两个客户端，完成一次聊天
4. 停止捕获

### 分析

1. 在捕获列表中找到 WebSocket 帧（Protocol 列显示 `WebSocket`）
2. 选中一条 `chat_message` 类型的帧
3. 展开 WebSocket 协议层 → 查看 Payload
4. 确认 payload 中的 `wrapped_key`、`nonce`、`ciphertext` 均为 Base64 编码
5. 确认 payload 中**不存在**任何可识别的聊天明文

### 截图要点

- 截图应包含过滤条件、帧列表和展开的 payload 三部分
- 如果使用 `127.0.0.1`，选择 Loopback 接口；如果使用局域网 IP，选择对应的网卡
