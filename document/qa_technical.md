## 1. 服务端的角色与公钥分发

### Q1.1: 服务端到底做了什么？和加密有关系吗？

服务端 (`chat_server.py`) 是一个**纯 WebSocket 盲转发中继**，核心设计原则：

- **只做消息路由和转发**，绝不解密任何 payload 内容
- **不持有任何用户的私钥**，也不存储聊天历史
- 日志只记录消息类型、发送方、接收方和 **payload 长度**，不打印明文

服务端与加密本身**没有直接关系**——它看到的所有聊天消息都是密文，无法还原内容。

### Q1.2: 服务端负责分发公钥给其他人吗？

**是的。** 公钥分发通过以下流程实现：

1. **用户注册时上报公钥** — 客户端连接后发送 `register` 消息，携带自己的公钥 PEM 文本
2. **服务端存储公钥** — 服务端维护 `self.public_keys: dict[str, str]` 字典（`user_id → 公钥PEM`）
3. **user_list 广播** — 每当有用户上线或下线，服务端向所有在线用户广播 `user_list` 消息，内容为 `{user_id: public_key_pem, ...}` 的完整映射
4. **客户端自动导入** — 收到 `user_list` 后，客户端自动将其中不认识的公钥导入本地缓存

> 注意：服务端只是**中转公钥文本**，它本身并不使用这些公钥做任何操作。

### Q1.3: 可以自己手动导入其他用户的公钥吗？

**可以，但目前主要依靠自动导入。**

- **桌面端 GUI** 提供了"加载私钥"和"导出公钥"按钮，但**没有提供"导入对方公钥文件"的 UI 按钮**
- **底层接口已支持** — `session_manager.set_peer_public_key(peer_id, pem_text)` 可以接受任意 PEM 公钥文本并导入
- **实际场景**中，连接服务器后 `user_list` 广播会自动完成公钥分发，用户不需要手动操作

如果要实现完全离线的公钥交换（不依赖服务端），只需调用 `set_peer_public_key` 接口即可，代码层面已经支持。

### Q1.4: 服务端能不能看到聊天内容？

**不能。** 服务端收到的 `chat_message` 中只有 `wrapped_key`（RSA 加密的 AES 密钥）、`nonce` 和 `ciphertext`（AES-GCM 密文），全部是 Base64 编码的密文。服务端没有任何用户的私钥，无法解密 `wrapped_key` → 无法得到 AES 密钥 → 无法解密 `ciphertext`。

## 2. 密钥存储与导入导出

### Q2.1: 私钥和公钥目前是怎么存的？

**分两种情况：**

| 端                  | 存储方式                                                                       | 持久化                                                                 |
| ------------------- | ------------------------------------------------------------------------------ | ---------------------------------------------------------------------- |
| **桌面端** (Python) | 运行时存储在内存中（`RSAKeyManager` 对象的 `private_key` / `public_key` 属性） | 可通过"导出公钥"保存为 `.pem` 文件；可通过"加载私钥"从 `.pem` 文件恢复 |
| **Web 端** (JS)     | 存储在浏览器内存的 `state.keyPair` 中（`CryptoKey` 对象）                      | **页面刷新即丢失**，不做持久化存储                                     |

### Q2.2: 密钥可以导出吗？

**桌面端可以，Web 端当前不提供导出 UI。**

桌面端提供以下功能：

| 功能           | UI 按钮    | 格式                                  |
| -------------- | ---------- | ------------------------------------- |
| 生成密钥对     | "生成密钥" | RSA-2048，内存存储                    |
| 从文件加载私钥 | "加载私钥" | PEM 文件（PKCS8 格式）                |
| 导出公钥到文件 | "导出公钥" | PEM 文件（SubjectPublicKeyInfo 格式） |

底层 `rsa_core.py` 还提供了更多导出接口：

- `serialize_private_key()` → PEM 字符串
- `serialize_public_key()` → PEM 字符串
- `save_private_key(file_path)` → 写入文件
- `save_public_key(file_path)` → 写入文件

Web 端的密钥在生成时设置了 `extractable: true`，技术上可导出，但 UI 中没有提供导出按钮。

### Q2.3: 密钥用的什么格式？

- **私钥**: PEM 编码，PKCS8 格式，无密码保护
- **公钥**: PEM 编码，SubjectPublicKeyInfo (SPKI) 格式
- **传输格式**: 公钥以 PEM 文本字符串形式通过 JSON 传输

示例：

```
-----BEGIN PUBLIC KEY-----
MIIBIjANBgkqhkiG9w0BAQEFAAOCAQ8AMIIBCgKCAQEA...
-----END PUBLIC KEY-----
```

## 3. 混合加密机制

### Q3.1: 大段内容是用 AES 加密的吗？

**是的。** 无论是文本消息还是文件内容，实际数据都由 **AES-256-GCM** 加密。

### Q3.2: RSA 只是用来分发 AES 密钥的吗？

**是的。** RSA 的唯一用途是**加密（包裹）一次性 AES 会话密钥**，即 RSA-OAEP 密钥封装。

完整的加密链路：

```
发送方:
  1. 生成随机 AES-256 会话密钥（os.urandom，32 字节）
  2. 用 AES-256-GCM 加密明文 → 得到 ciphertext + nonce
  3. 用接收方的 RSA-2048 公钥（OAEP + SHA-256）加密 AES 密钥 → 得到 wrapped_key
  4. 将 wrapped_key + nonce + ciphertext 打包发送

接收方:
  1. 用自己的 RSA-2048 私钥解密 wrapped_key → 得到 AES 会话密钥
  2. 用 AES-256-GCM + nonce 解密 ciphertext → 得到明文
```

### Q3.3: 每条消息都用不同的 AES 密钥吗？

**是的。** `message_crypto.encrypt_chat_message()` 每次调用都会执行 `aes_core.generate_aes_key(256)` 生成全新的随机密钥。这意味着：

- 即使某条消息的 AES 密钥泄露，也不影响其他消息
- 提供了一定程度的前向安全性

### Q3.4: 为什么不直接用 RSA 加密所有内容？

| 维度         | RSA 直接加密               | RSA + AES 混合加密（本项目）      |
| ------------ | -------------------------- | --------------------------------- |
| 速度         | 极慢（非对称运算）         | 快（AES 硬件加速）                |
| 明文长度限制 | RSA-2048 最多加密 190 字节 | 无限制                            |
| 实际消息处理 | 需要分块，复杂且慢         | AES 一次性处理，高效              |
| 业界实践     | 不推荐                     | TLS、PGP、Signal 等均采用混合方案 |

### Q3.5: AES-GCM 具体参数是什么？

| 参数       | 值                               |
| ---------- | -------------------------------- |
| 算法       | AES-GCM (Galois/Counter Mode)    |
| 密钥长度   | 256 位                           |
| Nonce 长度 | 12 字节 (96 bits)                |
| 认证标签   | 128 位（GCM 内置）               |
| 密钥生成   | `os.urandom()` — 操作系统 CSPRNG |

GCM 模式同时提供**机密性**和**完整性保护**（AEAD），如果密文被篡改，解密时会抛出 `InvalidTag` 异常。

## 4. 数字签名与消息认证

### Q4.1: RSA 有用来做数字签名（验证对方身份）吗？

**没有。** 当前系统中 RSA **仅用于加密/解密**（密钥封装），**没有数字签名功能**。

`rsa_core.py` 和 `message_crypto.py` 中都没有 `sign()` / `verify()` 相关代码。

### Q4.2: 那怎么确认消息确实来自对方？

当前系统的消息认证依赖于：

1. **协议层的 `sender_id` 字段** — 由发送方声明身份
2. **RSA 加密的隐式认证** — 只有持有正确私钥的接收方才能解密 `wrapped_key`，间接证明解密者是密钥持有人
3. **AES-GCM 的完整性保护** — 密文被篡改会导致解密失败

**但不能防止身份伪造** — 如果攻击者知道某人的公钥，可以用该公钥加密一条消息并伪造 `sender_id`。这是一个已知的设计简化。

### Q4.3: 如果要加数字签名，需要改哪里？

需要在 `message_crypto.py` 的加密流程中增加：

1. 发送方用自己的**私钥**对消息摘要签名
2. 将签名附加到协议消息中
3. 接收方用发送方的**公钥**验证签名

这是一个可扩展方向，但当前课程设计中不做此要求。

## 6. 安全性相关

### Q6.1: 系统的整体安全等级如何？

**优点：**

- 端到端加密，服务端无法读取明文
- AES-256-GCM 提供 AEAD（认证加密），防篡改
- 每条消息独立密钥，有一定前向安全性
- RSA-2048 + OAEP + SHA-256 密钥封装，符合现代最佳实践

**已知局限：**

- 无数字签名 → 无法防止身份伪造
- 无证书体系 → 公钥分发依赖服务端（可能被中间人替换）
- 私钥无密码保护 → PEM 文件导出后是明文
- Web 端密钥不持久化 → 页面刷新密钥丢失

### Q6.2: 如果服务端被攻破，会怎样？

攻击者控制服务端后：

- **能做的**：
  - 查看谁在和谁通信（流量分析）
  - 阻断消息传递（拒绝服务）
  - 替换新上线用户的公钥（中间人攻击，但已上线用户不受影响）
- **不能做的**：
  - 解密已传输的历史消息
  - 解密正在传输的消息（没有私钥）

### Q6.3: `os.urandom()` 生成密钥安全吗？

安全。`os.urandom()` 调用的是操作系统的密码学安全伪随机数生成器（CSPRNG）——Windows 上是 `CryptGenRandom`，Linux 上是 `/dev/urandom`。这是 Python 中生成密钥的标准做法。

## 8. 文件传输

### Q8.1: 文件传输也是加密的吗？

**是的。** 文件传输使用和文本消息完全相同的混合加密方案：

1. 生成一次性 AES-256 密钥
2. AES-GCM 加密文件内容
3. RSA-OAEP 加密 AES 密钥
4. 将 wrapped_key + nonce + ciphertext + 文件元数据一起发送

### Q8.2: 文件大小有限制吗？

有两种传输模式：

| 模式     | 条件       | 协议消息类型                                                                    |
| -------- | ---------- | ------------------------------------------------------------------------------- |
| 整体传输 | 文件 ≤ 5MB | `file_transfer`                                                                 |
| 分块传输 | 文件 > 5MB | `file_chunk`（按块发送，每块包含 `transfer_id`, `chunk_index`, `total_chunks`） |

### Q8.3: 文件内容在传输中能被还原吗？

不能。文件内容经过 AES-256-GCM 加密后以 Base64 编码传输，Wireshark 抓包只能看到密文。只有持有对应 RSA 私钥的接收方才能解密。

## 9. 浏览器安全上下文与 WebSocket

### Q9.1: 为什么浏览器会限制 Web Crypto API 只能在 HTTPS 或 localhost 下使用？

因为浏览器把 Web Crypto API、摄像头、麦克风、地理位置等能力都视为高风险接口，只允许在“安全上下文”里开放。安全上下文的目标不是限制开发者，而是避免普通 HTTP 页面在不可信网络里悄悄调用敏感能力、泄露密钥，或者在中间人攻击下被篡改后继续执行。

本项目的 Web 端需要生成 RSA 密钥对、导入公钥、执行 AES-GCM 加密和 RSA-OAEP 包裹密钥，这些都依赖 `crypto.subtle`。如果页面不是 HTTPS，也不是浏览器认定的本机地址，浏览器就会拒绝这些 API。

### Q9.2: 为什么 localhost / 127.0.0.1 可以例外？

因为浏览器把本机回环地址当作开发调试例外。`localhost` 和 `127.0.0.1` 的流量不会真正离开本机，浏览器默认认为它们的风险比远程 HTTP 小，所以即使不是 HTTPS，也允许访问 Web Crypto API。

这也是为什么项目在本机调试时可以直接打开 `http://localhost:8080`，但把同一份页面放到远程服务器后，就必须改成 `https://<服务器IP>/`。

### Q9.3: ws 和 wss 有什么区别？

`ws://` 是明文 WebSocket，相当于 WebSocket 版的 HTTP；`wss://` 是通过 TLS 加密的 WebSocket，相当于 WebSocket 版的 HTTPS。

| 协议     | 是否加密 | 是否校验完整性 | 典型场景               |
| -------- | -------- | -------------- | ---------------------- |
| `ws://`  | 否       | 否             | 本机调试、临时内网测试 |
| `wss://` | 是       | 是             | 正式部署、远程访问     |

如果一个页面是 HTTPS，浏览器通常不会允许它再去连接 `ws://`，因为这会形成 mixed content。换句话说，页面是“安全的”，但 WebSocket 却是“明文的”，浏览器会直接拦截这种降级连接。

### Q9.4: 这个限制可以关闭吗？

**不能在正式部署中关闭。** 这是浏览器运行时的安全模型，不是我们项目代码里的开关。它的设计目的就是阻止不安全页面随意调用敏感 API，或者让 HTTPS 页面偷偷退回到明文 WebSocket。

开发或测试时可以用这些方式解决：

1. 继续使用 `localhost` / `127.0.0.1` 本地调试。
2. 给服务器部署 HTTPS，再通过 `wss://` 连接。
3. 使用自签名证书或内部 CA 做内网测试。

但不建议把浏览器的安全限制当成长期关闭方案；正式部署最好保持 HTTPS + WSS。

### Q9.5: 本项目里是怎么处理这个问题的？

`web/app.js` 已经做了自动检测：如果当前页面不是 `localhost` 或 `127.0.0.1`，就会根据页面协议自动拼出 `wss://<host>/ws`；本机调试时则保留普通 `ws://`。

也就是说：

- 本机开发：`http://localhost:8080` + `ws://127.0.0.1:8765`
- 远程部署：`https://<服务器IP>/` + `wss://<服务器IP>/ws`

这不是两套不同的协议栈，而是同一套 WebSocket 连接在不同安全上下文下的正确用法。
