## 2. 三阶段目标

三阶段的唯一主目标，是把当前已经具备的密码模块、协议层、服务端和客户端网络层，真正收敛为一个可直接演示、可稳定验收的桌面端到端加密聊天程序。

三阶段完成后，项目应达到以下状态：

1. 两个桌面客户端可以连接同一服务端并看到彼此在线。
2. 发送方在 GUI 中输入文本后，可以完成加密并发送密文消息包。
3. 接收方收到消息后可以自动解密并显示明文。
4. GUI 能明确展示连接状态、联系人、公钥指纹和加解密过程日志。
5. 项目达到 NECESSARY_REQUIREMENT.md 所要求的最低合规演示标准。

## 4. 三阶段开发顺序

### 任务 A：收口桌面端启动与连接主流程

目标：确保用户从“打开程序”到“进入可聊天状态”的流程稳定、清晰、可重复。

涉及模块：

- desktop_chat_gui.py
- chat_client.py
- session_manager.py

必须完成：

- 固定登录区的最小输入项，只保留服务端地址、用户 ID、连接与断开操作。
- 固定密钥区的使用顺序，确保先生成或加载密钥，再连接服务端。
- 固定连接成功、连接失败、断开连接三种状态下的界面提示。
- 固定按钮启用与禁用逻辑，避免重复连接、断连后状态残留等问题。
- 固定程序关闭时的资源释放逻辑，避免后台线程或连接残留。

完成标准：

- 单个客户端可以稳定完成“生成密钥 -> 连接服务端 -> 断开 -> 再连接”的闭环。
- 状态栏与按钮状态始终与实际连接状态一致。

### 任务 B：收口联系人、公钥与会话状态

目标：确保桌面端在联系人选择、公钥同步和会话切换时行为一致，不出现“能看见联系人但无法加密”的半可用状态。

涉及模块：

- desktop_chat_gui.py
- session_manager.py

必须完成：

- 固定在线用户列表刷新逻辑，确保联系人上下线时界面同步更新。
- 固定自动导入对方公钥的主路径，同时保留失败提示。
- 固定当前联系人切换逻辑，确保状态栏或标签能反映当前聊天对象。
- 固定公钥指纹展示位置，便于演示时人工核验。
- 明确“未获取对方公钥时禁止发送”的提示策略。

完成标准：

- 选择任意在线联系人后，界面可以清晰显示当前会话对象和其公钥指纹。
- 未建立可用公钥状态时，发送链路不会进入错误的半执行状态。

### 任务 C：收口消息发送、自动解密与聊天显示

目标：把“发送密文消息包、接收端自动解密显示”真正稳定成课程演示主链路。

涉及模块：

- desktop_chat_gui.py
- session_manager.py
- message_crypto.py

必须完成：

- 固定发送前校验逻辑，包括空消息、未连接、未选择联系人、缺少对方公钥等情况。
- 固定发送成功后的本地消息显示规则。
- 固定接收消息后的自动解密流程，确保明文进入聊天区而不是只停留在日志区。
- 固定解密失败时的界面反馈，避免用户误以为消息丢失。
- 固定聊天区追加消息的格式，包括时间、发送方和系统消息区分。

完成标准：

- 客户端 A 发送文本后，客户端 B 可以自动显示解密后的明文。
- 解密失败时，聊天区和日志区都能明确提示异常。

### 任务 D：完善 Crypto Console 与安全状态展示

目标：让桌面端不仅“能聊天”，还能够在答辩时直观展示加密过程和安全设计。

涉及模块：

- desktop_chat_gui.py
- message_crypto.py
- session_manager.py

必须完成：

- 固定发送侧日志口径，至少包含原文长度、AES 会话密钥生成、wrapped_key 生成、密文发送成功。
- 固定接收侧日志口径，至少包含收到密文、wrapped_key 长度、RSA 解包成功、AES 解密成功。
- 明确区分系统日志、发送日志、接收日志和错误日志。
- 固定状态栏或界面中的安全信息展示，例如当前联系人、公钥指纹、连接状态。
- 保证日志内容适合截图展示，但不泄露不必要的敏感原文细节。

完成标准：

- 发送和接收各自至少形成一条完整的可视化加解密链路。
- 单看 GUI 右侧日志即可解释“为什么服务端看到的是密文”。

### 任务 E：完成最低合规版人工验收闭环

目标：把当前程序从“开发可运行”提升为“可重复演示、可课堂验收”。

涉及模块：

- tests/manual_acceptance.md
- desktop_chat_gui.py
- chat_server.py

必须完成：

- 固定本地启动服务端的步骤。
- 固定启动两个桌面客户端的步骤。
- 固定一次完整发送与接收演示步骤。
- 固定服务端盲转发验证步骤，明确检查日志中没有明文。
- 固定异常演示步骤，例如未选联系人、未导入公钥、连接断开、解密失败提示。
- 固定截图清单，为后续报告和汇报做准备。

完成标准：

- 按文档步骤可以稳定复现一次完整的双客户端加密聊天演示。
- 验收步骤不依赖额外口头补充解释。

### 任务 F：完成三阶段注释与可运行性审查

目标：确保桌面主程序已经满足 NECESSARY_REQUIREMENT.md 中“充分注释”和“程序可运行”的红线要求。

涉及模块：

- desktop_chat_gui.py
- chat_client.py
- session_manager.py
- tests/manual_acceptance.md

必须完成：

- 审查 GUI 事件函数是否具备足够注释和 docstring。
- 审查网络线程与主线程交互是否有清晰说明。
- 审查加密、解密、状态同步和异常处理路径是否有解释性注释。
- 审查手工运行步骤是否足够让他人在本地独立复现。

完成标准：

- 抽查关键模块时，仅看注释即可理解主流程。
- 新同学按文档即可运行出最低合规版演示结果。

## 6. 三阶段验收标准

当以下条件全部满足时，三阶段可以结束，并视为“最低合规版桌面演示程序”已经完成：

1. 服务端可以启动，两个桌面客户端可以连接同一服务端。
2. 两个客户端都可以看到在线用户列表中的对方。
3. 发送方可以在 GUI 中向当前联系人发送文本消息。
4. 接收方收到后可以自动解密并显示明文。
5. 服务端日志中不出现任何聊天明文。
6. GUI 中可以清晰看到加密和解密过程日志。
7. 异常场景下界面能给出明确提示。
8. 手工验收文档可以独立复现实验演示。
