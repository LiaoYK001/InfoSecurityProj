## 2. 二阶段目标

本阶段只聚焦协议层和服务端联调层，不进入 Web 端、文件传输、打包和展示包装。

二阶段完成后，项目应达到以下状态：

1. 客户端与服务端之间的消息格式统一且可稳定序列化。
2. 服务端能够完成注册、在线用户维护、消息盲转发和断线清理。
3. 客户端网络层能够稳定接收服务端消息，并向上层提供事件回调或事件队列。
4. 加密消息、回执、心跳、公钥同步等基础消息类型可以协同工作。
5. 为三阶段桌面 GUI 联调准备好稳定的网络与协议基础。

## 3. 二阶段范围

### 3.1 本阶段要完成的内容

- 固化消息协议，统一字段与消息类型。
- 完善服务端中继逻辑，确保只转发、不解密。
- 完善在线用户、公钥和心跳的服务端维护策略。
- 整理客户端网络层的收发事件模型。
- 建立协议和服务端的自动化测试或最小联调测试路径。
- 补充人工验收步骤，确保后续 GUI 可以直接接入。

### 3.2 本阶段明确不做的内容

- 不做 Web 客户端。
- 不做文件传输。
- 不做 Wireshark 展示材料整理。
- 不做 Windows 打包。
- 不做最终报告、PPT 和提交包整理。
- 不做群聊、历史记录、复杂账号系统。

## 4. 二阶段开发顺序

### 任务 A：收敛消息协议

目标：把客户端与服务端之间的 JSON 消息结构固定下来，避免后续联调时字段不一致。

必须完成：

- 统一消息类型集合，至少覆盖 register、public_key、chat_message、ack、error、heartbeat、user_list。
- 统一消息公共字段，至少包括 type、sender_id、receiver_id、timestamp、payload。
- 明确各类消息的 payload 结构，避免不同模块各自解释。
- 约定缺省值规则，例如 receiver_id 缺省为空字符串、timestamp 由构造端自动填充。
- 约定服务端对消息的最小校验规则，只做结构校验和路由判断，不解析密文业务内容。

完成标准：

- 所有网络消息都能稳定序列化与反序列化。
- 非法 JSON、缺字段、未知类型都能被明确拒绝。

### 任务 B：固化中继服务端行为

目标：将服务端职责限制在“注册、维护连接、转发消息、广播在线状态”四件事上。

必须完成：

- 建立客户端注册流程，注册时记录用户 ID、连接对象与公钥信息。
- 建立按 receiver_id 路由的盲转发机制。
- 建立在线用户列表维护与广播机制。
- 建立公钥同步或转发机制，支持客户端自动获取对方公钥。
- 建立掉线清理逻辑，避免断线后在线表残留。
- 建立最小日志策略，只记录消息类型、发送方、接收方和长度，不记录明文内容。
- 建立错误消息返回机制，方便客户端提示用户。

完成标准：

- 两个已注册客户端之间可以互相转发消息。
- 服务端日志不包含任何聊天明文。
- 某一端离线后，服务端可以及时清理状态。

### 任务 C：完善心跳与连接状态

目标：让连接状态可预测、可提示，避免客户端界面出现“看似在线、实际已断开”的情况。

必须完成：

- 约定心跳消息格式与发送频率。
- 约定服务端对心跳的处理方式，只刷新状态，不做业务计算。
- 约定客户端断线、重连、超时的统一提示方式。
- 约定服务端对非活跃连接的清理策略。

完成标准：

- 客户端可以识别连接成功、连接断开、发送失败、目标离线等状态。
- 服务端能够在连接断开后同步更新在线列表。

### 任务 D：整理客户端网络事件模型

目标：为三阶段 GUI 接入准备稳定的网络接口，不让界面直接依赖底层 WebSocket。

必须完成：

- 统一客户端接收事件的类型表达。
- 统一聊天消息、公钥消息、在线列表、错误、回执的事件分发方式。
- 统一发送队列与接收队列的使用边界。
- 统一异常上报方式，避免网络线程直接影响 UI 线程。

完成标准：

- 上层调用者可以用统一事件模型处理所有网络反馈。
- 网络收发逻辑与界面逻辑保持解耦。

### 任务 E：完成协议与服务端联调测试

目标：证明协议与服务端的闭环已经稳定可用。

必须完成：

- 构造注册消息并验证服务端注册行为。
- 构造聊天密文消息并验证服务端盲转发行为。
- 构造公钥消息并验证客户端之间的同步逻辑。
- 构造非法消息并验证服务端拒绝与错误反馈。
- 构造断线场景并验证清理逻辑。

完成标准：

- 协议层和服务端层能通过最小联调验证。
- 关键路径在异常情况下也能给出明确反馈。

## 5. 二阶段建议优先级

### 第一优先级：先把协议和中继闭环固定

- 消息协议统一。
- 服务端注册与转发统一。
- 在线列表与公钥同步统一。

### 第二优先级：把状态和异常做稳

- 心跳。
- 掉线清理。
- 错误消息返回。
- 客户端事件模型。

### 第三优先级：为 GUI 联调做准备

- 最小联调脚本。
- 手工验收步骤。
- 调试日志口径统一。

---

## 6. 二阶段验收标准

当以下条件全部满足时，二阶段可以结束并进入三阶段桌面界面联调：

1. 客户端注册后，服务端能够正确维护在线状态。
2. 服务端能够按接收方 ID 正确转发密文消息。
3. 客户端能够接收并识别聊天消息、公钥消息、错误消息和在线列表消息。
4. 非法消息可以被协议层和服务端层正确拦截。
5. 连接断开后，服务端状态能够正确清理。
6. 客户端网络接口已经足以支撑后续 GUI 直接调用。

---

## 7. 二阶段输出物

二阶段结束时，至少应形成以下可交付成果：

- 一份稳定的消息协议规范。
- 一个可盲转发的服务端中继方案。
- 一套可供 GUI 调用的客户端事件模型。
- 一份协议与中继的最小验收记录。
- 一份面向三阶段的联调准备说明。
