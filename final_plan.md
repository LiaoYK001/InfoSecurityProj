### 1.2 底线版最终定义

为了保证课程要求一定达标，本项目的“最低合规版”定义如下：

1. 使用 Python 实现。
2. 使用 AES-GCM 对聊天消息正文进行加密与解密。
3. 使用 RSA-OAEP 对 AES 会话密钥进行交换。
4. 有一个服务端和两个桌面客户端。
5. 客户端 A 可以向客户端 B 发送密文消息。
6. 服务端只转发数据，不解密。
7. 客户端 B 收到后自动解密并显示明文。
8. 核心源码具有充分注释。
9. 程序可以稳定运行并完成演示。
10. 报告、汇报材料和最终提交物完整。

说明：

- Web 端不是 NECESSARY_REQUIREMENT.md 的硬性要求，它属于增强项。
- 文件传输不是 NECESSARY_REQUIREMENT.md 的硬性要求，它属于增强项。
- Wireshark 抓包证明不是硬性要求，但非常适合作为加分项，保留在增强阶段。

## 4. 可直接开做的详细开发任务清单

以下任务按“必须先做的保底版”和“后做的增强版”拆分。只有前一项完成并验收后，才进入下一项。

## 4.1 P0 保底达标版

### 任务 A：整理并改造 rsa_core.py

目标：保留现有 RSA 能力，转为“密钥管理 + 会话密钥交换”的底层模块。

文件：rsa_core.py

必须完成的内容：

- [ ] 保留 RSA 密钥对生成、保存、加载、导出 PEM 的能力。
- [ ] 保留对方公钥导入能力。
- [ ] 保留 encrypt_session_key_for_peer 和 decrypt_session_key_from_peer 这类会话密钥交换接口。
- [ ] 新增公钥指纹函数，例如 get_public_key_fingerprint()，用于界面展示和人工核验。
- [ ] 明确注释：RSA 直接加密长消息仅保留为教学演示，不作为聊天消息主链路。
- [ ] 所有公开方法补全 docstring，说明输入、输出和用途。

完成标准：

- [ ] 能生成本地 RSA 密钥对。
- [ ] 能导出公钥给对方。
- [ ] 能用对方公钥加密 AES 会话密钥。
- [ ] 能用本地私钥解密 AES 会话密钥。

### 任务 B：新增 AES 消息加密模块

目标：实现聊天消息正文的实际加密算法，满足 NECESSARY_REQUIREMENT 对消息加密算法的要求。

文件：aes_core.py

必须完成的内容：

- [ ] 实现 generate_aes_key()，生成 128 bit 或 256 bit AES 密钥。
- [ ] 实现 encrypt_text(plaintext: str, key: bytes) -> dict。
- [ ] 实现 decrypt_text(cipher_payload: dict, key: bytes) -> str。
- [ ] 使用 AES-GCM，输出至少包含 nonce、ciphertext。
- [ ] 统一 Base64 编码规则，保证网络传输时字段可序列化。
- [ ] 对空消息、错误密钥、篡改数据给出明确异常。
- [ ] 每个关键步骤加注释，说明为何使用 AES-GCM 以及它如何保证机密性和完整性。

建议输出结构：

```json
{
  "nonce": "base64_nonce",
  "ciphertext": "base64_ciphertext"
}
```

完成标准：

- [ ] 同一条消息加密后不能直接看到明文。
- [ ] 使用正确密钥可以稳定解密。
- [ ] 篡改密文后解密必须失败。

### 任务 C：新增混合加密封装模块

目标：把 RSA 和 AES 串成完整链路，生成真正用于聊天消息的加密包。

文件：message_crypto.py

必须完成的内容：

- [ ] 实现 encrypt_chat_message(plaintext, peer_public_key, sender_key_manager)。
- [ ] 内部流程固定为“生成 AES 会话密钥 -> AES 加密正文 -> RSA 加密 AES 密钥 -> 组包返回”。
- [ ] 实现 decrypt_chat_message(message_payload, local_private_key)。
- [ ] 输出统一字典结构，供 chat_protocol.py 直接使用。
- [ ] 增加可视化所需的调试字段生成逻辑，例如原文长度、密文长度、密钥指纹。
- [ ] 给关键步骤写注释，确保之后写报告时能直接解释实现原理。

建议输出结构：

```json
{
  "wrapped_key": "base64_wrapped_key",
  "nonce": "base64_nonce",
  "ciphertext": "base64_ciphertext"
}
```

完成标准：

- [ ] 发送方能生成完整加密包。
- [ ] 接收方能用本地私钥和 AES 密钥链路还原原文。

### 任务 D：新增消息协议模块

目标：把网络消息统一成可序列化、可扩展、可调试的格式。

文件：chat_protocol.py

必须完成的内容：

- [ ] 定义消息类型常量：register、public_key、chat_message、ack、error、heartbeat。
- [ ] 定义统一字段：type、sender_id、receiver_id、timestamp、payload。
- [ ] 实现 make_register_message()。
- [ ] 实现 make_public_key_message()。
- [ ] 实现 make_chat_message()。
- [ ] 实现 parse_message() 和基础字段校验。
- [ ] 每种消息结构写清楚注释，方便后续报告中的“详细设计”章节直接引用。

建议聊天消息结构：

```json
{
  "type": "chat_message",
  "sender_id": "alice",
  "receiver_id": "bob",
  "timestamp": "2026-04-15T12:00:00Z",
  "payload": {
    "wrapped_key": "...",
    "nonce": "...",
    "ciphertext": "..."
  }
}
```

完成标准：

- [ ] 所有网络消息都能以 JSON 字符串稳定序列化与反序列化。
- [ ] 非法消息能及时被识别并报错。

### 任务 E：新增服务端中继模块

目标：实现“只转发，不解密”的服务端，满足即时通信基础功能。

文件：chat_server.py

必须完成的内容：

- [ ] 实现客户端连接接入。
- [ ] 实现用户注册，维护 user_id -> 连接对象 的映射。
- [ ] 实现公钥上传和缓存。
- [ ] 实现消息按 receiver_id 路由转发。
- [ ] 实现心跳保活与掉线清理。
- [ ] 服务端日志只记录连接、消息类型、发送方、接收方、消息长度。
- [ ] 显式保证服务端不解密 payload，不打印明文，不保存私钥。
- [ ] 所有核心函数写注释，说明为什么服务端是盲转发。

完成标准：

- [ ] 两个客户端都能连接到服务端。
- [ ] A 发给 B 的消息可以通过服务端转发到 B。
- [ ] 服务端日志中看不到聊天明文。

### 任务 F：新增客户端网络层模块

目标：把 GUI 和网络解耦，避免后续界面改动影响通信稳定性。

文件：chat_client.py

必须完成的内容：

- [ ] 实现 connect() 和 disconnect()。
- [ ] 实现 register(user_id, public_key)。
- [ ] 实现 send_chat_message(receiver_id, encrypted_payload)。
- [ ] 实现 receive_loop()，持续监听服务器消息。
- [ ] 设计回调接口给 GUI 使用，例如 on_message、on_error、on_status_change。
- [ ] 增加发送失败、掉线、重连异常处理。
- [ ] 用注释说明网络线程与 GUI 主线程之间的数据流。

完成标准：

- [ ] GUI 不直接操作 socket/WebSocket 底层对象。
- [ ] 收发消息可通过回调更新界面。

### 任务 G：新增会话管理模块

目标：统一管理本地密钥、对方公钥、会话状态和最近联系人。

文件：session_manager.py

必须完成的内容：

- [ ] 保存本地 RSAService 或 KeyManager 实例。
- [ ] 保存 peer_id -> peer_public_key 的映射。
- [ ] 提供 load_local_keys()、export_local_public_key()。
- [ ] 提供 set_peer_public_key() 和 get_peer_public_key()。
- [ ] 提供 encrypt_for_peer() 和 decrypt_from_peer() 的统一入口。
- [ ] 提供安全状态摘要，例如当前联系人、是否已建立安全通道、公钥指纹。
- [ ] 用注释写清楚会话状态与密钥状态之间的关系。

完成标准：

- [ ] GUI 不直接拼接加密链路，只调用 session_manager。

### 任务 H：新增桌面聊天主界面

目标：形成真正的课程主程序，满足“即时通信软件基础功能”。

文件：desktop_chat_gui.py

必须完成的内容：

- [ ] 登录区：输入 user_id、服务器地址、连接按钮。
- [ ] 密钥区：生成本地密钥、加载私钥、导出公钥、导入对方公钥。
- [ ] 联系人区：输入对方 user_id 或选择当前联系人。
- [ ] 聊天区：消息列表、输入框、发送按钮。
- [ ] 状态区：显示连接状态、当前联系人、公钥指纹。
- [ ] 收到聊天消息后自动解密并显示明文。
- [ ] 发送前必须先做 AES 加密和 RSA 会话密钥封装。
- [ ] 发送失败、解密失败、未导入对方公钥时弹窗或状态提示。
- [ ] 右侧增加 Crypto Console，打印加密和解密过程日志。
- [ ] 所有 GUI 事件函数写注释，清楚说明触发逻辑。

最小演示流程：

1. 客户端 A 连接服务器。
2. 客户端 B 连接服务器。
3. A/B 各自加载或生成 RSA 密钥。
4. A 导入 B 的公钥。
5. A 向 B 发送一条消息。
6. B 自动解密并在界面显示明文。

完成标准：

- [ ] 双桌面客户端文本聊天链路跑通。
- [ ] 发送链路中界面显示加密日志。
- [ ] 接收链路中界面显示解密日志。
